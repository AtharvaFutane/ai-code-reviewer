Phase 1 (hours 0–6): understand before you build anything
The single most important instruction in the whole brief is "read the entire brief before writing a single line of code," and the same applies to the diffs themselves.

Extract the assignment zip and open all three diff files. Read them like a human reviewer would, not like you're about to feed them to a model. As you read, jot down — anywhere, a scratch file, a notes app — every line that looks suspicious to you: unparameterized SQL, a loop that calls the database once per item, a missing null check, anything that smells wrong. You're not trying to be exhaustive or correct here, you're building your own private intuition for what "a good finding" looks like, which becomes your personal sanity check later when your agent's actual output comes back. This step is easy to rush past and it's the cheapest leverage in the whole 48 hours.
Sign up at console.groq.com and get an API key, drop it into your (gitignored) .env. Do a bare-minimum sanity check that a basic call to the model actually returns text — you're not building anything yet, just confirming the door is open.
Design your state schema, on paper or in a markdown file, not in code yet. Concretely: what's the one shared object that flows through every node in your graph? At minimum it needs to hold the incoming diff text, language, and optional context, plus somewhere for each of the five agents to drop its own list of findings without stepping on the others, plus a slot for the final merged report once that's ready. Deciding this shape now saves you from reshuffling five agents' worth of code later.
Scaffold the FastAPI app itself — just enough that uvicorn starts and a placeholder root route responds. No real logic yet.

Commit checkpoint: once the venv works, a basic Groq call returns text, and FastAPI boots, commit that as something like "phase 1: project setup, Groq connectivity confirmed, state schema drafted." Push.

Phase 2 (hours 6–20): build the pipeline, one agent at a time
This is the core of the assignment and where most of your 48 hours should go. The instruction worth taking seriously here is to build and test one agent at a time against Diff 1, rather than writing all five and debugging them simultaneously — five things failing at once is much harder to reason about than one.

For each of the five reviewers in turn (security, performance, correctness, style, test coverage): write that agent's instructions so it only looks for its own category of issue, run it against Diff 1 alone, and check the output actually parses into a clean list of findings with real line numbers before moving to the next agent. The security one is worth building first since the brief explicitly wants it "paranoid," and getting one agent fully right gives you a template to copy and adapt for the other four rather than starting from a blank page each time.
Commit after each agent works, individually, on Diff 1. Five small commits here ("add security reviewer agent," "add performance reviewer agent," and so on) tell a much clearer story than one commit that says "agents done," and if something breaks later you can bisect to exactly which agent introduced it.
Once all five produce sane output independently, build the merge node: it has to collect all five findings lists, catch cases where two agents flagged the same or adjacent lines and decide whether that's one issue described two ways or genuinely two issues, assign an overall severity (a reasonable default: any critical finding makes the whole thing critical), and decide the verdict. A defensible starting rule — though the brief is explicit that there's no single correct architecture — is: any critical finding means request_changes; only low-severity style or test-coverage notes means approve; anything ambiguous in between means needs_discussion.
Wire the actual graph last: an entry point that fans out to all five agent nodes, edges from all five into the merge node so the graph waits for every agent before merging, then a clean exit. This is the part the rubric calls "LangGraph architecture," and it's evaluated as its own line item — a clean fan-out/fan-in is worth more than something that quietly runs the agents one after another and calls it a pipeline.

Phase 3 (hours 20–32): put it behind the API and run it for real

Implement POST /review: it takes the diff, language, and optional context, feeds them into the compiled graph, and returns whatever the merge node produced. Store each result in memory (a dictionary keyed by a generated review id is enough) so the lookup endpoints in Phase 4 have something to read from.
Before building the full ingestion script, sanity-check the endpoint by hand with a quick curl or Postman request using a tiny made-up diff — you don't want to debug the endpoint and the script at the same time.
Write the local script that reads each of the three .txt diffs from disk, posts each to your running endpoint, and saves the response into your reviews/ folder as diff1_review.json, diff2_review.json, diff3_review.json. This script is itself graded, so it should actually run cleanly, not just exist.
Run all three diffs through and read the output against the notes you took in Phase 1. This is where most of your score actually comes from, so the iteration here matters more than almost anything else: tighten prompts where findings are vague or generic, add detail where line numbers are wrong, and rein in any agent — security especially — that's flagging things that aren't really problems, since false positives cost points just as much as missed bugs do.

Commit checkpoint: "implement POST /review and local ingestion script," then a follow-up commit or two as you tighten prompts: "improve agent prompts after testing against all three diffs."
Phase 4 (hours 32–42): round out the API and make it not break

Add the remaining three endpoints: GET /review/{id} to fetch one stored result, GET /reviews to list summaries of everything generated this session, and GET /health to confirm the service and Groq are both up.
Handle the unglamorous failure cases: what happens on an empty or missing diff, what happens if Groq times out or errors, and — this is the one people skip — what happens if an agent's raw output doesn't parse cleanly into your expected structure. LLM output occasionally comes back malformed; the request shouldn't crash because of it, it should degrade gracefully.
Re-run all three diffs one more time through the finished pipeline and confirm the three JSON files sitting in your reviews/ folder are current and correct, not leftovers from an earlier, rougher version of your prompts.

Commit checkpoint: "add remaining endpoints, error handling, finalize reviews output."
Phase 5 (hours 42–48): document, record, ship

Read your three JSON outputs one more time, critically, as if you were the senior engineer being asked to trust them. Would you actually agree with these findings? This is your last chance to catch something embarrassing before a human compares your output to the answer key.
Write the README: one command to start the project, the architectural decisions you made and why (state schema shape, how the merge node decides verdicts, why you split agents the way you did), and honestly, what you're most and least happy with. If you used AI tools significantly while building this, say so briefly — the brief says they're curious, not suspicious about it.
Make sure .env.example exists at the root with the required key names (no real values).
Record the two-minute demo: show the server starting, run one diff through live (the script or a single API call works), and show the JSON it produces. Keep it tight.
Final commit, push, and double check the repo is actually public and that everything — the three review JSONs, the script, the README, .env.example — is actually sitting in the repo and not just on your machine.
